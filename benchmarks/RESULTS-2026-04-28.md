# E2E benchmark — 2026-04-28

Cluster: AKS gb300, 17× GPU nodes, 8× NVMe per node, NDR InfiniBand (UCX/RDMA).
Dataset: `nvidia_DeepSeek-R1-0528-NVFP4-v2/` — 350 files, 6 465 chunks, 413.34 GB.
Image: `sha-54c9742-arm64`. Helm rev 13.

## Workflow per run

1. `deploy/wipe-caches.sh` — wipe every node's `/mnt/nvme/blobcache-cache`.
2. `kubectl rollout restart ds/blobcache-blobcached` — fresh daemon state.
3. `deploy/storage-access.sh on` — open Azure storage public access.
4. `POST /hydrate` to one pod with `chunk_concurrency=32`, peers fan-out via HRW.
5. PASS1 — every pod sequentially reads the full 413 GB tree (cold cache, peer-served).
6. PASS2 — every pod re-reads (warm cache, local NVMe).
7. `deploy/storage-access.sh off`.

Run #4 and #5 within 0.6 % on total wall-time.
**35 % faster total than last-good baseline (872 s).**
**22× faster hydrate vs last-good baseline (359 s).**

## Recommended config (committed in `deploy/helm/blobcache/values.yaml`)

```yaml
cache.chunkSize:               4194304   # 4 MiB
azure.blockSize:               4194304   # 4 MiB
azure.workers:                 8
azure.poolMaxIdlePerHost:      512
transport.chunkConcurrency:    32
transport.peerConcurrency:     32
transport.prefetchDepth:       0         # disabled
transport.prefetchConcurrency: 4         # ignored when depth=0
```

## Key findings

- **prefetch_depth=16 starves foreground reads** of the 32 chunk-permits → PASS1 1055 s.
  Disabling: −30 % (1055 → 741 s).
- **64 MiB chunks cause peer-fetch head-of-line blocking.**
  4 MiB chunks: additional −56 % on PASS1 (741 → 307 s).
  Hydrate unaffected (Azure-bandwidth-bound either way).
- **Per-shard hydrate tail spread** collapsed from ~1.3× → 1.05× with 4 MiB chunks.
- **Peer transport scales linearly** — single wiped pod (run #3) reads cold
  data via 16 peers in 584 s, matching the per-pod mean (552 s) of the
  17-pod-concurrent run #4. Zero Azure GETs, zero errors.
- **Brief Azure throttle** at hydrate fan-out start (~12 retry_sleep_s/s,
  zero giveups, status 200/206) — harmless, clears within 1 minute.

## Run #6 — TCP transport (apples-to-apples vs RDMA)

Same helm config as runs #4/#5 (4 MiB chunks, prefetch_depth=0,
chunk_concurrency=32, peer_concurrency=32, azure.workers=8) — only
`config.transport.kind` flipped from `rdma` -> `tcp`. Same image
(`sha-54c9742-arm64`, built `--features ucx`; the tcp transport is
in-tree at `src/transport.rs` and selected at runtime). Helm rev 14.

| Phase | TCP (run #6) | RDMA (run #5) | TCP / RDMA |
|---|---|---|---|
| HYDRATE wall | 16.33 s | 15.64 s | 1.04x |
| HYDRATE aggregate BW | 24.15 GiB/s | 24.61 GiB/s | 0.98x |
| PASS1 aggregate wall | 428.10 s | 307.0 s | **1.39x slower** |
| PASS1 per-pod mean | 396.3 s | ~354 s | 1.12x slower |
| PASS1 per-pod spread | 1.18x | 1.04x | wider tail |
| PASS2 aggregate wall | 231.82 s | 237.6 s | 0.98x (same) |
| PASS2 per-pod mean | 208.0 s | 222 s | 0.94x |
| TOTAL e2e wall | 676.25 s | 560.6 s | **1.21x slower** |

Read: TCP costs 21% on the full pipeline. The cost is concentrated in
PASS1 — the cold-cache, peer-fetch-dominated phase. PASS2 (warm,
local NVMe) is identical because the transport is not on that path,
and HYDRATE is identical because both transports are Azure-bandwidth-
bound, not peer-bound.

The PASS1 tail spread also widens (1.18x vs 1.04x), consistent with TCP
suffering more head-of-line blocking under the chunk_concurrency=32
fan-out than UCX RC.

### Bottom line

For this workload (350 files, 413 GB, 17 pods, NDR IB) RDMA buys ~21%
end-to-end and ~28% on cold-cache reads. RDMA wins are entirely on
the peer hot-path; if the cluster were CPU-bound elsewhere or the
fabric were Ethernet, the gap would shrink.
